# Moral I9 模拟器 — 修复问题总结

本文档记录在本项目中针对 **MStar I9（非 MT6252）** 固件所做的输入、时间与电量相关修复：**问题背景、处理过程、难点与涉及文件**。

---

## 一、背景与原始问题

模拟器基于早期 **MT6252** 方案演进，但 **Moral I9 实际跑的是 MStar I9 固件**，大量外设地址、中断号与键盘扫描方式与 MT6252 不一致。

用户侧曾反馈的典型现象包括：

| 现象 | 说明 |
|------|------|
| 黑壁纸 | 与显示管线（DE / DMA2D / 图层）相关，另有专项修复（花屏→正常显示） |
| 键盘无效 | 仍按 MT6252 写 `0x81070000` 一带扫描寄存器，固件根本不读 |
| 触摸无效 | 事件有触发，但坐标语义与固件期望的 ADC 范围不一致 |
| 时间卡在固定值（如 05:24） | RTC 寄存器未与主机时间对齐，且仅靠 VM 事件+中断时更新路径不可靠 |
| 电量不显示/异常 | 辅助 ADC 返回值与真实通道语义未对齐，未知通道未处理 |

**重要结论：** 用户确认 `FindKeypadData` / `FindRomMpuSettingAddr` / `FindNorFlashId` 等 **MT6252 专用探测逻辑对 Moral I9 不适用**，不应依赖其填充 `keypaddef` 等表项来解决键盘问题。

---

## 二、修复内容概览

### 1. 键盘（核心：换寄存器 + 换中断号 + 行列矩阵）

**问题根因**

- MT6252 路径：向 `KEYBOARD_REG_BASE (0x81070000)` 等写扫描位，并配合 `DEV_IRQ_CHANNEL_KEYBOARD = 8`。
- MStar I9：通过 **HAL** 读 **`0x34000814`** 作为键盘状态（`HalKeypadGetStatus`），数据在 **`0x34000804`**（`HalKeypadGetData`），清中断写 **`0x3400080C`**（`HalKeypadClearInt`）。
- 中断向量表中 **IRQ 12 对应 `DrvKeypadISR`**（与项目内 `中断函数表.txt` 一致），而不是 MT6252 使用的 8。

**实现要点**

- 按键按下：在 `0x34000814` 写入 **低电平有效** 的行列组合：行在低字节、列在高字节，对应位清零表示该行列被拉低。
- 松开：写回 **`0xFFFF`** 表示无键。
- 模拟器启动前将 `0x34000814` 初始化为 `0xFFFF`，避免 RAM 全 0 被误判为“全键按下”。
- SDL 键位索引 → 行列：依据固件中 **KEY_0…KEY_9、*、#、SIDE_UP/DN、SK、SEND** 等字符串表附近的扫描码结构，推出 **scan = row×8 + col**，建立 `mstarKeyMap[]`（见 `src/keypad.h`）。
- VM 事件里触发中断改为 **`KPD_IRQ_LINE = 12`**，并与 `StartInterrupt` 的掩码位一致（固件 `HalKeypadIrqInit` 会 unmask 对应位）。

**涉及文件**

- `src/keypad.h`：MStar 寄存器宏、映射表、`KPD_IRQ_LINE`。
- `src/keypad.c`：`SimulatePressKey` / `handleKeyPadVmEvent` 重写。
- `src/main.c`：启动前写 `0x34000814 = 0xFFFF`；`FindKeypadData()` 改为空实现（去掉对已删除 `keypaddef` 的依赖）。

---

### 2. RTC 时间

**问题根因**

- 仅依赖 **VM_EVENT_RTC_IRQ + `StartInterrupt(RTC)`** 时才 `Update_RTC_Time()`，若掩码/中断时机不满足，寄存器长期保持 NAND/镜像里旧值（表现为固定时间）。
- 即使用户曾尝试在 `main` 里写 RTC，若与“周期性更新”未打通，仍会停滞。

**实现要点**

- **启动前**：用 `time()` / `localtime()` 写 `RTC_SECOND_REG`～`RTC_YEAR_REG`（与固件“年从 2000 起”的约定一致：`tm_year - 100`）。
- **运行中**：在 `RtcTaskMain()` 中 **每次定时入队 RTC 事件前先调用 `Update_RTC_Time()`**，保证寄存器与主机时间同步，不完全依赖中断是否成功投递。

**涉及文件**

- `src/rtc.c`：`RtcTaskMain()` 内增加 `Update_RTC_Time()`。
- `src/main.c`：启动线程前初始化 RTC 寄存器块。

---

### 3. 触摸（像素 → ADC 量程）

**问题根因**

- 固件触摸路径里既有 **AUX 触摸命令读数**，也有 **`0x3400C1C0` / `0x3400C1C8` 等寄存器**（与 TS ADC 完成中断相关，见 `中断函数表.txt`：**29 = HalTSAdcDoneIrqHandler，31 = HalTSPenDetIrqHandler**）。
- 若直接把 **屏幕像素坐标** 写入，与校准/驱动期望的 **ADC 满量程（约 0～1023）** 不一致，上层会认为触点非法或不变。

**实现要点**

- `touchscreen.c` 中 X/Y 读命令返回：  
  `X_adc = touchX * 1023 / 240`，`Y_adc = touchY * 1023 / 400`（与当前模拟分辨率 240×400 对应，可按实机校准再调）。
- `vmEvent.c` 中 ADC 采样中断分支：对打包的 x/y 同样做缩放后再写入 `0x3400C1C0` / `0x3400C1C8`。

**涉及文件**

- `src/touchscreen.c`
- `src/vmEvent.c`

---

### 4. 电池 / 辅助 ADC

**问题根因**

- 电池或杂项通道通过 **AUX_TS_CMD** 选择后读 **`AUX_TS_DATA1`**；原逻辑对未知命令只打印，未给合理采样值。
- `AUX_UnkReg (0x82050010)` 曾写 `0xFFF`，易落在“一格电”或饱和区，与满电 UI 预期不符。

**实现要点**

- 未知通道：向 `AUX_TS_DATA1` 写入试探值 **`0x350`**（作为较合理的“中高电量”ADC 占位，仍需结合固件电量曲线精调）。
- `AUX_BASE` 分支里同步将 `AUX_UnkReg` 改为与上述策略一致的占位值。

**难点说明**

- 固件中存在 **“BatteryInfo not implement”** 类字符串时，部分 UI 可能根本不读真实 ADC，此时模拟器只能尽量“喂合理值”，无法保证 100% 与真机一致。

**涉及文件**

- `src/touchscreen.c`

## 三、过程与方法

1. **区分平台**  
   明确 Moral I9 ≠ MT6252，删除/停用 MT6252 专用 ROM 探测与 `keypaddef` 方案，避免“看似修了键盘、实际固件不读”的假修复。

2. **IDA 静态分析**  
   - 搜索并反编译 `DrvKeypadISR`、`HalKeypadGetStatus`、`HalKeypadGetData`、`HalKeypadClearInt`、`HalKeypadIrqInit` 等，得到 **真实 MMIO 地址与中断注册参数**。  
   - 从 ROM 中 **KEY_*** 字符串表周边字节结构解析出行列与扫描码关系，建立 SDL 键索引到 (row,col) 的映射。

3. **与本地文档交叉验证**  
   使用仓库内 **`中断函数表.txt`**：确认 **12 → DrvKeypadISR**，**29/31 → 触摸 ADC / 笔检测**，与代码中 IRQ 选择一致。

4. **编译与迭代**  
   在 MinGW 环境下 `make` 通过；去除 `keypaddef` 后补齐 `FindKeypadData` 空实现，消除链接/编译错误。

## 四、难点与风险

| 难点 | 说明 |
|------|------|
| **遗留 MT6252 与 MStar 混用** | 同一仓库里 LCD/中断/键盘地址容易抄错；必须以 **反编译符号 + 中断表** 为准。 |
| **IDA 交叉引用工具不稳定** | 部分 `xrefs`/立即数搜索对 Thumb 拼接地址不敏感，需结合 **函数名列表（如 list_funcs *key*）** 与 **get_bytes 读表**。 |
| **键盘矩阵语义** | 低电平有效、行列 bitmask、与 `gnRowMask/gnColMask` 组合后的行为需与 `DrvKeypadISR` 一致，否则出现“全按下”或“无反应”。 |
| **触摸校准未知** | 线性映射 1023/240、1023/400 是工程近似；真机若有非线性校准表，需进一步对照固件。 |
| **电量可能未完整实现** | 字符串提示未实现时，UI 行为取决于上层逻辑，模拟器侧只能提供占位 ADC。 |
| **IRQ 掩码与 StartInterrupt** | 若固件尚未 unmask 对应位，`StartInterrupt` 会失败；键盘侧依赖启动后 `HalKeypadIrqInit` 正常执行。 |

## 六、后续可优化方向

- 根据 IDA 中 **电量曲线表** 或 **具体 AUX 通道号** 替换 `0x350` 占位逻辑。  
- 对照固件 **触摸校准结构体**，改为非线性映射或从镜像读取校准参数。  
- 若 **壁纸仍为黑**：单独排查 DE 多层合成、背景层 buffer 是否参与 `0x7400313C` 路径（与本文输入/RTC/电量修复相互独立）。  
- 软键左右键若硬件映射不同，可根据实机或反汇编细化 `mstarKeyMap` 中 SKLEFT/SKRIGHT 的行列（当前曾暂用同一列占位）。

---

*文档对应工程状态：键盘按 MStar `0x34000814` + IRQ12；RTC 启动初始化 + 周期 `Update_RTC_Time`；触摸与 TS ADC 寄存器写 ADC 量程；电池相关为 AUX 占位增强。*

## 七、补充（2025-03）：为何「修了仍无效」——外设区未映射

若界面仍显示 **05:24**、**电池空**、触摸无反应，请先查看控制台：

1. **`[mem] Mapped 0x81000000..0x83000000`**  
   若未出现，说明外设池映射失败。
2. **`[RTC] init … write_err=0 readback_hour=与当前小时一致`**  
   若 `write_err≠0` 或 `readback_hour` 为乱值，说明 RTC 写入仍未落到可读写内存。
3. **大量 `地址无法访问:810xxxxx` / `820xxxxx`**（`hookRamErrorBack`）  
   表示 CPU 或主机侧访问了 **未 `uc_mem_map` 的地址**。

**根因：** `RTC_*`（`0x810b****`）、`AUX_*` 触摸（`0x8205****`）、`GPT`（`0x8106****`）等均在 **0x81000000～0x83xxxxxx** 一带，而 `initMtkSimalator` 原先只映射了 ROM、RAM、`0x34000000`、`0x74000000`、`0x90000000` 等，**没有映射 0x81000000**。  
后果：`uc_mem_write(RTC_…)` 静默失败（或走未映射钩子）、`hookRam` 对 `0x82050000` 的模拟**根本不会触发**（Unicorn 仅在已映射且可访问的地址上触发 MEM 钩子）。

**修复：** 增加 `uc_mem_map_ptr(MTK, 0x81000000, 0x02000000, …)`（32MB 外设占位池），并在启动时打印 `[RTC] init …` 做回读校验。

**RTC 事件队列：** 原先 `VM_EVENT_RTC_IRQ` 在 `StartInterrupt` 失败时会反复 `EnqueueVMEvent`，可能占满队列；已改为仅尝试投递中断，时间刷新仍由 `RtcTaskMain()` 中的 `Update_RTC_Time()` 负责。

**壁纸仍为黑：** 与上述外设映射无关时，需单独查 DE 图层/资源加载路径。

## 八、`bin/log.txt` 实例：映射成功却仍 05:24

日志若出现：

- `[mem] Mapped 0x81000000..0x83000000`
- `[RTC] init … write_err=0 readback_hour=17`（与当时主机小时一致）

说明 **0x810b 已能读写**，但状态栏仍 **05:24**，原因是：**界面时钟来自 `HalRtcGetSecondCount` → `HalAsuraRead`（SPI Asura），不是 `0x810b`。**

**修复：** 在 `HalRtcGetSecondCount` 写出秒计数前（`STR R5,[R0]`，PC≈`0x31b9c`）用 `hookCodeCallBack` 将 **R5** 覆盖为 `moral_fw_rtc_seconds_since_2000()`（本地 `mktime(2000-01-01 00:00:00)` 至当前的秒差 + **`config.h` 中 `MORAL_RTC_SECOND_OFFSET`**）。  
若出现「比宿主机慢 16 小时」（如 17:18 显示 01:18），典型在 `config.h` 设 `MORAL_RTC_SECOND_OFFSET` 为 **`57600`**（16×3600）；已对齐则改为 **`0`**。

**触摸：** 日志中 `handle touch down/up` 与 **`KER_VERROR_DIAGNOSE R0:8000918`** 同时出现，表示事件已进 VM，但固件触摸路径仍断言/报错，需单独查 TS IRQ 与寄存器语义。
