# 待修改项目

本文档用于整理模拟器当前待修复与待完善事项，优先聚焦以下核心问题：

- LCD 刷新异常与显示稳定性
- 整体运行流畅性（卡顿、忙循环、任务饥饿）
- SD 卡初始化链路
- SIM 卡初始化链路

---

## 一、优先级与推进策略

建议按以下顺序推进，避免问题互相干扰：

1. **先修“基础时序与调度”**：定时器、IRQ 投递、事件队列节流。
2. **再修“显示链路”**：DE/LCD 完成状态、刷新触发条件、区域更新策略。
3. **然后修“外设初始化”**：SD 与 SIM 的寄存器返回值、状态机阶段、超时处理。
4. **最后做“联调与压测”**：长时间运行、频繁触摸、页面切换、开机初始化路径复测。

---

## 二、LCD 刷新问题（高优先级）

### 现象

- 局部刷新不稳定，可能出现刷新延迟、闪烁、卡住不更新。
- 不同 UI 页面切换时，刷新节奏不一致。
- 小区域（如光标/触摸反馈）与大区域更新逻辑行为差异明显。

### 重点排查项

1. **DE 完成标志与 IRQ 时机**
   - 检查 DE 触发后，完成状态写回是否可靠。
   - 检查“先置完成状态再触发 IRQ”与“先 IRQ 再置状态”对固件 busy-loop 的影响。

2. **LCD 刷新触发条件**
   - 核对是否存在“某状态位必须置位才允许刷屏”的隐含条件（如电源状态/睡眠状态）。
   - 检查是否存在“仅事件触发刷新，缺少兜底周期刷新”的场景。

3. **区域刷新坐标与 pitch**
   - 校验 `Lcd_Update_X/Y/W/H`、`pitch`、源地址映射是否一致。
   - 检查边界裁剪是否导致写屏区域错位或被截断。

4. **小区域刷新节流策略**
   - 当前节流避免了死循环，但可能造成视觉延迟。
   - 需要确定稳定帧率目标（例如 30fps 或 60fps）并统一策略。

### 建议改进

- 建立统一“刷新提交函数”，所有 DE/LCD 路径统一走一套状态机。
- 加入最小监控日志（可开关）：触发时间、区域、完成时间、IRQ 结果。
- 增加“周期性保底刷新”（低频）防止事件丢失导致界面静止。

### 验收标准

- 连续触摸拖动、页面切换 3 分钟内无明显闪烁、无长时间停帧。
- 刷新延迟可控（主观流畅、无明显掉帧）。
- 不再出现 LCD busy-loop 造成的 CPU 占满。

## 三、整体流畅性问题（高优先级）

### 现象

- 模拟器有时明显卡顿，任务响应慢。
- 某些事件可能高频堆积，影响主循环吞吐。

### 重点排查项

1. **事件队列压力**
   - 统计 `VM_EVENT_*` 入队/出队速率。
   - 排查触摸 MOVE、定时 IRQ 是否出现“生产速度 > 消费速度”。

2. **忙等待与死循环**
   - 排查 `while(1)`、高频 polling、无 sleep 的循环。
   - 对调试阶段保留的阻塞行为进行隔离（debug 开关控制）。

3. **中断重投递策略**
   - 明确“重投递上限”和“退避策略”，避免中断风暴。
   - 对失败重投递增加节拍限制，避免每个 basic block 都尝试。

4. **日志输出开销**
   - 高频路径日志应默认关闭或采样打印。
   - 避免 I/O 打印拖慢实时路径。

### 建议改进

- 增加性能统计面板（每秒输出一次）：队列长度、IRQ 次数、渲染次数、平均帧间隔。
- 将高频事件（尤其 MOVE）做统一节流与去重。
- 对关键线程明确“让出 CPU”点，避免长期独占。

### 验收标准

- 同样操作路径下，CPU 占用和卡顿显著下降。
- 事件队列无持续增长趋势，不出现长期积压。
- 开机后持续运行 10 分钟不出现“越跑越卡”。

## 四、SD 卡初始化问题（高优先级）

### 现象

- 固件侧可能识别不到 SD 卡，或初始化流程卡在某阶段。
- 文件系统挂载失败、读写异常、超时返回等问题可能存在。

### 重点排查项

1. **控制器寄存器语义**
   - 命令发送、响应读取、状态位（忙/完成/错误）是否符合固件预期。
   - 中断状态与清除逻辑是否正确。

2. **初始化时序**
   - 上电、复位、卡检测、时钟配置、总线宽度切换步骤是否完整。
   - CMD0/CMD8/ACMD41/CMD2/CMD3/CMD7 等关键阶段返回值是否合理。

3. **镜像与容量信息**
   - CSD/CID、块大小、容量等信息是否与镜像一致。
   - 读写边界处理是否正确（越界、对齐、长度）。

4. **错误码与超时**
   - 区分“无卡”“超时”“CRC 错误”“非法命令”等返回。
   - 固件是否依赖特定错误码进入降级路径。

### 建议改进

- 先实现“最小可用初始化路径”，保证识别与基础读扇区稳定。
- 再补全错误场景与异常恢复流程。
- 为 SD 初始化阶段增加分段日志（阶段号 + 关键寄存器快照）。

### 验收标准

- 稳定识别 SD 卡并完成文件系统挂载。
- 基础文件读写稳定，无随机超时。
- 多次重启后初始化成功率稳定。

## 五、SIM 卡初始化问题（中高优先级）

### 现象

- SIM 可能初始化失败，导致网络相关功能不可用。
- ATR/复位流程、状态位返回可能与固件预期不一致。

### 重点排查项

1. **上电与复位流程**
   - 冷复位/热复位时序是否满足固件状态机要求。
   - SIM 插入状态检测与去抖逻辑是否有效。

2. **ATR 与协议参数**
   - ATR 返回格式、长度、时序是否正确。
   - 波特率/协议协商（如 T=0）是否至少满足基础分支。

3. **状态寄存器与中断**
   - 发送完成、接收可读、错误状态位语义是否匹配。
   - 中断置位与清除时机是否正确。

4. **超时与重试**
   - 固件重试路径是否能被正确触发并最终收敛。
   - 避免“无响应但状态看似正常”的假成功。

### 建议改进

- 先做“可通过初始化检查”的最小模型，再逐步补齐 APDU 细节。
- 对 SIM 相关寄存器访问建立 trace（可开关），便于定位状态机卡点。

### 验收标准

- 初始化阶段稳定通过，不再频繁掉入错误分支。
- 与网络栈联动时，不因 SIM 状态异常阻塞主流程。

---

## 六、建议新增的基础保障

1. **统一调试开关**
   - 以宏或配置控制日志等级，避免频繁改代码开关日志。

2. **关键路径断言与保护**
   - 对关键状态（空指针、非法地址、越界参数）做轻量保护，防止级联故障。

3. **回归测试清单（手工）**
   - 开机流程、触摸拖动、页面切换、SD 挂载读写、SIM 初始化、长稳运行。

4. **版本化问题追踪**
   - 每次修复记录“问题现象/根因/改动点/验证结果”，避免回归时重复排查。

---

## 七、短期迭代计划（建议）

### 第 1 轮（先稳）

- 收敛定时与事件投递节奏，压制中断风暴。
- 修正 LCD 刷新触发与完成时序，确保界面稳定更新。

### 第 2 轮（补初始化）

- 打通 SD 初始化最小闭环（识别 + 读扇区 + 挂载）。
- 打通 SIM 初始化最小闭环（上电/复位/ATR 通过）。

### 第 3 轮（提体验）

- 优化帧率稳定性与输入响应。
- 增加长稳运行验证与回归脚本（可选）。

---

## 八、当前建议先执行的 4 个动作

1. 增加每秒性能统计输出（事件队列长度、IRQ 次数、刷新次数）。
2. 梳理并固定 LCD 刷新状态机（触发→完成→IRQ→下一帧）。
3. 补 SD 初始化分段日志，确认失败阶段（命令/响应/状态位）。
4. 补 SIM 初始化分段日志，确认卡在上电、ATR 还是状态位判定。
