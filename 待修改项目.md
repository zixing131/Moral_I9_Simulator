# 待修改项目

本文档用于整理模拟器当前待修复与待完善事项，优先聚焦以下核心问题：

- LCD 刷新异常与显示稳定性
- 整体运行流畅性（卡顿、忙循环、任务饥饿）
- SD 卡初始化链路
- SIM 卡初始化链路

---

## 零、本轮修复结果（2026-03-24）

本轮已完成以下代码级修复：

1. **基础时序与调度修复**
   - 已补上 `GptTaskMain()` 与 `SimTaskMain()` 的主循环调度，此前二者定义存在但未实际执行。
   - 已为高频 VM 事件增加去重合并，避免 `LCD/MSDC/DMA/GPT/SIM/RTC` 类事件在队列中重复堆积。
   - 已增加每秒性能统计输出：队列长度、高水位、入队/出队、丢弃/合并、IRQ 次数、LCD 刷新次数、SD 读写次数。

2. **LCD / 流畅性修复**
   - 已将 LCD 周期刷新、DE 触发、FMark、layer/buffer format 等热路径日志改为默认关闭，减少 I/O 对主循环的拖慢。
   - 已统计实际 LCD 刷新次数，便于后续观察是否仍有“高频空刷新”。

3. **SD 链路修复**
   - 已将 SD 读操作优先使用已缓存的镜像大小，避免频繁重复探测文件长度。
   - 已将 SD 写入上限从“仅 1GB 硬上限”修正为“优先以实际镜像大小为上限”，防止写越界把镜像扩坏。
   - 已支持越界截断写与越界告警，避免初始化或文件系统写元数据时直接把镜像写坏。
   - 已将高频 SD I/O 日志改为默认关闭，减少启动阶段大量块读写日志对吞吐的影响。
   - 已补上基于可执行文件目录的 SD 镜像回退查找，启动时可稳定自动加载 bin/Rom 下镜像。
   - 已修正 FCIE CMD17/CMD18/CMD24/CMD25 的扇区参数解码，避免把非地址字段错误拼成高 16 位，导致启动阶段出现 GB 级越界读。

4. **SIM 链路修复**
   - 已在初始化时补写 SIM 插卡状态、状态寄存器、卡类型寄存器，避免固件始终看到“无卡/未就绪”。
   - 已补充 `READ BINARY` / `READ RECORD` 最小响应路径。
   - 已补齐多组 `SELECT + GET RESPONSE` 缺失响应，覆盖 `6FAE/6FC6/6FC5/6F43/6F4A/6F4B/6F4C` 等文件。
   - 已为 SIM DMA 返回数据增加长度保护，避免短响应被按更长长度直接越界拷贝。

5. **验证状态**
   - 已完成 `make` 编译，当前修改可以通过构建。
   - 已定位并修复一个启动级崩溃：`InitSimCard()` 中新增的 `uc_mem_write` 发生在 `uc_open` 与外设映射之前，导致程序白屏后立即退出。
   - 修复后已确认进程可持续运行，并已拿到新的 `[perf]` 统计输出，说明事件循环、LCD 刷新统计和后台任务调度已经实际生效。
   - 已确认启动日志不再出现“没有SD卡镜像文件，跳过加载”，当前会稳定打印已使用的镜像路径与容量信息。
   - 已确认此前启动阶段连续出现的 SD 越界读日志已消失；当前短时运行未再复现该问题。
   - 当前仍需继续在本机界面环境下观察 SDL 窗口内容，并继续验证更深层的 SD 挂载结果与 SIM 初始化推进情况。

## 一、优先级与推进策略

建议按以下顺序推进，避免问题互相干扰：

1. **先修“基础时序与调度”**：定时器、IRQ 投递、事件队列节流。
2. **再修“显示链路”**：DE/LCD 完成状态、刷新触发条件、区域更新策略。
3. **然后修“外设初始化”**：SD 与 SIM 的寄存器返回值、状态机阶段、超时处理。
4. **最后做“联调与压测”**：长时间运行、频繁触摸、页面切换、开机初始化路径复测。

## 二、LCD 刷新问题（高优先级）

### 当前状态

- **部分已修复**：热路径日志已默认关闭，减少了 DE/LCD 相关 I/O 开销。
- **部分已修复**：已增加 LCD 刷新计数，便于确认是否仍有空转刷新。
- **仍需运行验证**：由于本轮未拿到新的运行日志，尚不能确认闪烁、停帧是否已完全消失。

### 现象

- 局部刷新不稳定，可能出现刷新延迟、闪烁、卡住不更新。
- 不同 UI 页面切换时，刷新节奏不一致。
- 小区域（如光标/触摸反馈）与大区域更新逻辑行为差异明显。

### 重点排查项

1. **DE 完成标志与 IRQ 时机**
   - 检查 DE 触发后，完成状态写回是否可靠。
   - 检查“先置完成状态再触发 IRQ”与“先 IRQ 再置状态”对固件 busy-loop 的影响。

2. **LCD 刷新触发条件**
   - 核对是否存在“某状态位必须置位才允许刷屏”的隐含条件（如电源状态/睡眠状态）。
   - 检查是否存在“仅事件触发刷新，缺少兜底周期刷新”的场景。

3. **区域刷新坐标与 pitch**
   - 校验 `Lcd_Update_X/Y/W/H`、`pitch`、源地址映射是否一致。
   - 检查边界裁剪是否导致写屏区域错位或被截断。

4. **小区域刷新节流策略**
   - 当前节流避免了死循环，但可能造成视觉延迟。
   - 需要确定稳定帧率目标（例如 30fps 或 60fps）并统一策略。

### 建议改进

- 建立统一“刷新提交函数”，所有 DE/LCD 路径统一走一套状态机。
- 加入最小监控日志（可开关）：触发时间、区域、完成时间、IRQ 结果。
- 增加“周期性保底刷新”（低频）防止事件丢失导致界面静止。

### 验收标准

- 连续触摸拖动、页面切换 3 分钟内无明显闪烁、无长时间停帧。
- 刷新延迟可控（主观流畅、无明显掉帧）。
- 不再出现 LCD busy-loop 造成的 CPU 占满。

## 三、整体流畅性问题（高优先级）

### 当前状态

- **已修复核心问题**：事件队列已支持高频事件去重，能直接压制 IRQ/LCD/SIM/MSDC 类重复入队。
- **已修复核心问题**：SIM/GPT 后台任务已接回主更新线程，不再长期“状态机不推进但外围重复等待”。
- **已修复辅助问题**：新增每秒性能统计，后续可以直接从日志观察是否仍有队列持续增长。
- **仍待确认**：`PHY2MIU` 相关异常路径是否仍是开机阶段的主要 CPU 热点，需结合新运行日志继续看。

### 现象

- 模拟器有时明显卡顿，任务响应慢。
- 某些事件可能高频堆积，影响主循环吞吐。

### 重点排查项

1. **事件队列压力**
   - 统计 `VM_EVENT_*` 入队/出队速率。
   - 排查触摸 MOVE、定时 IRQ 是否出现“生产速度 > 消费速度”。

2. **忙等待与死循环**
   - 排查 `while(1)`、高频 polling、无 sleep 的循环。
   - 对调试阶段保留的阻塞行为进行隔离（debug 开关控制）。

3. **中断重投递策略**
   - 明确“重投递上限”和“退避策略”，避免中断风暴。
   - 对失败重投递增加节拍限制，避免每个 basic block 都尝试。

4. **日志输出开销**
   - 高频路径日志应默认关闭或采样打印。
   - 避免 I/O 打印拖慢实时路径。

### 建议改进

- 增加性能统计面板（每秒输出一次）：队列长度、IRQ 次数、渲染次数、平均帧间隔。
- 将高频事件（尤其 MOVE）做统一节流与去重。
- 对关键线程明确“让出 CPU”点，避免长期独占。

### 验收标准

- 同样操作路径下，CPU 占用和卡顿显著下降。
- 事件队列无持续增长趋势，不出现长期积压。
- 开机后持续运行 10 分钟不出现“越跑越卡”。

## 四、SD 卡初始化问题（高优先级）

### 当前状态

- **已修复一部分根因**：镜像大小被缓存，读写边界按真实镜像大小约束，不再仅靠 1GB 回退上限。
- **已修复一部分根因**：启动时已可自动从可执行文件目录稳定找到并加载 SD 镜像，不再依赖碰运气的工作目录。
- **已修复一部分根因**：FCIE 路径的块地址解码已修正，启动阶段不再把命令附带字段拼成超大扇区号。
- **已修复一部分性能问题**：高频 SD I/O trace 默认关闭，减少启动阶段文件系统读写拖慢。
- **已拿到新的启动日志**：当前已能确认 SD 镜像被加载并打印容量，但还需继续验证文件系统是否完全挂载成功。
- **仍待继续补齐**：若启动后仍卡在 SD 初始化，需要继续补 `CMD3/CMD7/CMD13/ACMD51` 等寄存器语义与 FCIE 响应细节。

### 现象

- 固件侧可能识别不到 SD 卡，或初始化流程卡在某阶段。
- 文件系统挂载失败、读写异常、超时返回等问题可能存在。

### 重点排查项

1. **控制器寄存器语义**
   - 命令发送、响应读取、状态位（忙/完成/错误）是否符合固件预期。
   - 中断状态与清除逻辑是否正确。

2. **初始化时序**
   - 上电、复位、卡检测、时钟配置、总线宽度切换步骤是否完整。
   - CMD0/CMD8/ACMD41/CMD2/CMD3/CMD7 等关键阶段返回值是否合理。

3. **镜像与容量信息**
   - CSD/CID、块大小、容量等信息是否与镜像一致。
   - 读写边界处理是否正确（越界、对齐、长度）。

4. **错误码与超时**
   - 区分“无卡”“超时”“CRC 错误”“非法命令”等返回。
   - 固件是否依赖特定错误码进入降级路径。

### 建议改进

- 先实现“最小可用初始化路径”，保证识别与基础读扇区稳定。
- 再补全错误场景与异常恢复流程。
- 为 SD 初始化阶段增加分段日志（阶段号 + 关键寄存器快照）。

### 验收标准

- 稳定识别 SD 卡并完成文件系统挂载。
- 基础文件读写稳定，无随机超时。
- 多次重启后初始化成功率稳定。

## 五、SIM 卡初始化问题（中高优先级）

### 当前状态

- **已修复核心缺陷**：SIM 后台任务此前未被调度，本轮已接回主循环。
- **已修复最小状态模型**：已补插卡状态、卡类型和若干常见 APDU/SELECT 响应。
- **仍待运行验证**：需要新的开机日志确认是否已经从“初始化失败”进入“网络栈可继续推进”的阶段。
- **仍待继续补齐**：ATR 目前仍是简化模型，如果固件后续卡在 ATR/PTS 协商，需再补完整 ATR。

### 现象

- SIM 可能初始化失败，导致网络相关功能不可用。
- ATR/复位流程、状态位返回可能与固件预期不一致。

### 重点排查项

1. **上电与复位流程**
   - 冷复位/热复位时序是否满足固件状态机要求。
   - SIM 插入状态检测与去抖逻辑是否有效。

2. **ATR 与协议参数**
   - ATR 返回格式、长度、时序是否正确。
   - 波特率/协议协商（如 T=0）是否至少满足基础分支。

3. **状态寄存器与中断**
   - 发送完成、接收可读、错误状态位语义是否匹配。
   - 中断置位与清除时机是否正确。

4. **超时与重试**
   - 固件重试路径是否能被正确触发并最终收敛。
   - 避免“无响应但状态看似正常”的假成功。

### 建议改进

- 先做“可通过初始化检查”的最小模型，再逐步补齐 APDU 细节。
- 对 SIM 相关寄存器访问建立 trace（可开关），便于定位状态机卡点。

### 验收标准

- 初始化阶段稳定通过，不再频繁掉入错误分支。
- 与网络栈联动时，不因 SIM 状态异常阻塞主流程。
